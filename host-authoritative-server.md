# Host-Authoritative Co-op — SneakBit Online

The chosen online co-op design. One player is the **host** and runs the existing single-player game unchanged; up to three **guests** join the host's world via a thin relay. The same architecture covers desktop split-screen via a native wrapper that bundles the relay and spawns it on `127.0.0.1` — no special-case "local-only" code path. This document is the single authoritative spec for online co-op; nothing else (CLAUDE.md, README.md, code comments) overrides it.

## Confirmed agreements

1. **The host's browser is the entire simulation.** The host runs today's single-player code unmodified. Guests' avatars are added to that simulation the same way a second local player would be.
2. **The server is a relay + lobby, never a simulator.** It pairs hosts with guests by invite code, forwards frames, and dies as small as possible. No tick loop, no game logic.
3. **Game must feel fluid on every client.** Guests use **client-side prediction** for their own avatar and bullets, and **interpolation** for everything else. Local input has zero perceived latency.
4. **Client distribution stays on GitHub Pages** at <https://curzel.it/sneakbit-html>. The VPS hosts only the relay.
5. **Client has three modes** selected by URL: offline (default, no param), host (`?host=1`), guest (`?join=K7MJ2`). Each has separate save state from the others.
6. **Identity is an anonymous UUIDv4** generated by the browser on first run, persisted in `localStorage`. No usernames, no accounts, no auth.
7. **Sessions are formed by invite code** (e.g. `K7MJ2`, 5 alphanumeric chars). The host opens a session and gets a code; guests enter the code to join.
8. **In-memory only.** The server holds sessions and connection lists in process memory. A restart drops every session. No DB.
9. **Cheating prevention is out of scope.** The trust model is "friends playing together." A malicious guest can desync, send bogus inputs, or spam — but they can't write the world; only the host can.
10. **Session lifetime = host lifetime.** When the host disconnects, the session ends and guests are returned to single-player offline. No host migration.
11. **Native wrappers reuse the same network model.** An Electron/Tauri/Steam binary bundles the relay and the client; for local split-screen co-op it spawns the relay on `127.0.0.1` and opens N webviews pointing at the bundled `index.html` with role params. No new networking code, no new wire shapes.

## Why this exists

A fully-authoritative server (server simulates everything, clients only render) is robust but expensive: every entity, mob, and combat resolution runs on the VPS forever. A peer-ownership model (each entity has an owning client) is lighter but asks the codebase to grow an ownership broker, per-entity authority gates, and interaction routing. Host-authoritative is *lighter still* because the host runs the game **exactly as today**. There is no ownership map, no broker, no per-entity authority — there's the host's local simulation and nothing else. The guest is a thin renderer + input forwarder. The server is a 200-line relay.

The cost is that the host's lag is everyone's lag, and the host quitting ends the session. For "invite my friend to play with me for an hour" — the actual goal — both costs are fine.

## Vocabulary

- **Host** — the connected client whose simulation *is* the game. Owns every entity, every NPC, every pickup. Renders the world for itself locally.
- **Guest** — a connected client that sends inputs and renders state received from the host. Predicts its own avatar locally for fluidity.
- **Session** — a host + 0–3 guests sharing one running world. Lives until the host disconnects.
- **Invite code** — short, human-typable string (5 alphanumeric chars) that identifies a session for guests to join.
- **Relay** — the Node server. Routes frames between session members; runs no game logic.
- **Input intent** — a high-level command (`moveUp`, `interact`, `shoot`) the guest sends to the host. The host applies it as if it were a local-co-op P2/P3/P4 input.
- **Snapshot** — host-broadcast world state consumed by every guest.
- **Reconciliation** — guest's process of correcting its predicted avatar position when the host's snapshot disagrees.
- **Seq** — monotonic per-guest input sequence number. The host echoes the last-applied seq in every snapshot so the guest knows what to reconcile against.

## Architecture at a glance

```
host (browser)               relay                guest (browser)
─────────────                ─────                ──────────────
existing local tick   ◄───WS────  fan-out      ◄──── input intents (seq-stamped)
   (unchanged)        ─────WS───► fan-out      ────► snapshots (every tick)
                                                    │
                                                    ├──► render host's state
                                                    └──► reconcile own avatar against snapshot.lastSeq
```

- Single Node process. Maps sessions to connection lists; nothing else.
- WebSocket transport. JSON frames. Versioned handshake.
- No DB: all state is in process memory. Server restart = sessions dropped, hosts and guests go back to local play.
- **No `shared/` split required.** The client repo grows by one networking module + one prediction module on the guest side, and one input-injection module on the host side.

## Authority boundaries

| Concern | Owner |
|---|---|
| World state, NPCs, mobs, combat, pickups, gates, puzzles, dialogue, cutscenes, zone transitions | **Host** (always) |
| Guest's own avatar position, HP, inventory | **Host** is authoritative; guest predicts locally |
| Guest's input intents | **Guest** (the only thing they truly own) |
| Camera, zoom, animation interpolation, audio, HUD, menus, settings, key bindings | Each client locally |
| Save state | Host's localStorage. Guests do not persist session state. |
| Creative mode, map editor | Host only — disabled for guests. While the host is in creative / map editor, the "Start hosting" control is disabled with a tooltip directing them to leave creative first (no force-exit, no live-editing-with-guests in v0). |

The guest is allowed to **predict** but never to **decide**. If the guest's prediction disagrees with the host's snapshot, the host wins and the guest reconciles.

## Fluidity strategy

The single hard requirement of this design. Every input the user makes must produce a visible response within one frame (≤16 ms), even on a 200 ms-RTT guest.

### 1. Host fluidity — trivial
The host's game is unchanged. Input → local tick → local render. Zero network in the critical path. Identical to single-player today.

### 2. Guest fluidity — three techniques

**a) Client-side prediction (own avatar)**
- Guest stamps every input with a monotonic `seq` and sends it to the host.
- Guest *immediately* applies the input to a local copy of its own avatar (`predictedSelf`). The renderer draws `predictedSelf` for the guest's character — never the snapshot's lagged copy.
- Tile-locked stepping makes this dead simple: the only states are *idle*, *stepping toward (x±1, y±1)*. A predicted step either lands (host agrees) or snaps back (host says "wall").

**b) Server reconciliation**
- Every snapshot from the host includes `lastSeq[guestId]` — the highest input seq the host has applied for that guest.
- Guest keeps a ring buffer of unacknowledged inputs.
- On snapshot arrival:
  - Compare host's authoritative position for `self` against the position the guest predicted *as of `lastSeq`*.
  - If they match: drop acknowledged inputs from the buffer, keep going.
  - If they differ: snap `predictedSelf` to the host's position, then **re-apply** every input with seq > lastSeq to advance `predictedSelf` to "now."
- For tile-stepping this almost never triggers — corrections happen only on disputed steps (walls, doors guest doesn't know are closed).

**c) Entity interpolation (everything else)**
- Other players, NPCs, mobs, pushables, projectiles: guest renders by interpolating between the last two received snapshot frames.
- Hold-and-interpolate buffer: render at `now - 100 ms` so there's always a "next" frame to lerp toward. Tradeoff: 100 ms visual lag on non-self entities, in exchange for smoothness.
- If no new snapshot arrives for >300 ms, freeze and mark stale (subtle ghosting). >5 s → assume disconnect, return to single-player.

### 3. What the guest does *not* predict
- Damage outcomes ("did my bullet hit?"). Guest fires the bullet visually (predicted), but the host decides if it landed. Damage numbers and kill confirmations arrive one snapshot late — imperceptible.
- Pickup confirmations. Guest sees themself reach the item but the inventory increment waits for the host's snapshot. ~100 ms latency on the HUD update — acceptable.
- Zone transitions. Guest sees themself enter the teleporter, then the screen fades while waiting for the host to push the new zone snapshot.

Bottom line: guest *movement* and *spawning own bullets* are predicted; *world reactions* are not. This is the standard FPS recipe and it works.

## Identity

- On first run, the client generates a **UUIDv4** and persists it in `localStorage` under `sneakbit.online.uuid`. Sent on every WS connect.
- The server uses the UUID to recognize a returning client.
- Reconnect with the same UUID within 30 seconds → resume same player slot in the same session. Beyond 30 seconds → fresh session entry (guest must re-enter code; host must re-open).
- Display name: shortened UUID prefix (e.g. `Player-a3f9`). Custom names deferred.
- Multiple tabs with the same UUID → server closes the older connection with code 4003.

## Sessions and invites

A session is created by a host and joined by guests using an invite code.

- **Open a session (host):** client sends `host.open`. Server creates the session, generates a 5-char alphanumeric code, replies with `host.opened {code}`. Host displays the code in an HTML panel (pause menu → "Open to friends").
- **Join (guest):** client navigates to `?join=CODE` or types the code in the join panel. Client sends `guest.join {code}`. Server looks up the session, checks capacity (max 3 guests = 4 total), pairs them with the host, replies with `guest.joined {sessionId, hostName}` and tells the host via `peer.joined`.
- **Leave (guest):** explicit "Leave session" or WS close. Host is told via `peer.left` and removes the guest's avatar after the disconnect grace.
- **Close session (host):** host disconnects or clicks "Close to friends." Server broadcasts `session.closed` to every guest, who falls back to offline mode.
- **Session size cap:** 4 (1 host + 3 guests) in v0. Soft cap.
- **Session lifetime:** while the host is connected (with a 30 s reconnect grace). Once the host drops past grace, the session is GC'd.

There is no party model. A session *is* the group. Guests cannot promote, swap, or persist independently of a host.

### Deep-link `?join=CODE` while already in a session

If a `?join=CODE` URL is opened (or pasted into the URL bar) while the
client is already hosting or guesting, the deep-link is **honored
unconditionally**: the client auto-leaves its current session
(`host.close` if hosting — which ends co-op for all current guests —
or `guest.leave` if guesting) and then auto-joins the new session via
the code. The user sees the same flow as a fresh-launch deep-link.

Rationale: deep-links are typically followed deliberately (a friend
just shared one), and the alternative behaviors (silent ignore, modal
prompt) either trap the user in the wrong session or add a click on
the most common path. The cost — a host's existing guests get a
`session.closed` — is acceptable since hosting is also explicitly
initiated.

## Host setup

When the client boots in host mode (`?host=1`), it:
1. Runs the existing single-player game **identically** — same tick, same renderer, same save namespace.
2. On boot, generates/loads the UUID, opens a WS, sends `host.open`.
3. Receives `host.opened {code}` and displays it in the Party panel.
4. On `peer.joined`, spawns a guest avatar in the host's local world. This reuses the **existing local 2P co-op infrastructure**: a guest is `Player2` from the host's perspective (or P3/P4 for the third/fourth slot). The existing `coopMode.js`, `pickups.js`, `melee.js`, `shooting.js`, etc. already split inventory/equipment/HP per slot, so this is mostly already wired.
5. On every tick, serializes the world delta and broadcasts to all guests.
6. On every input intent received from a guest, injects it into the existing input pipeline as if it came from a local P2/P3/P4 device.
7. On `peer.left`, despawns the guest's avatar (after the 30 s reconnect grace) and frees the slot.

The host never sees a snapshot from anyone. Their local renderer is the truth.

## Guest setup

When the client boots in guest mode (`?join=CODE`), it:
1. **Skips its own simulation entirely.** No tick, no AI, no pickups — none of that happens locally.
2. Loads only the renderer, input layer, HUD, audio, asset loader.
3. Generates/loads the UUID, opens a WS, sends `guest.join {code}`.
4. Receives `guest.joined` and a `snapshot` for the host's current zone.
5. Maintains:
   - A **mirror world** built from received snapshots (every entity, including own avatar's authoritative copy).
   - A **predicted self** (own avatar's predicted position based on local inputs).
   - A **mirror buffer** with the last two snapshots for interpolation.
   - An **input log** of unacknowledged inputs for reconciliation.
6. Sends `input` to the host (via the relay) on every input edge, stamped with `seq`.
7. Renders: predicted self for own avatar, interpolated mirror for everything else.
8. Reconciles on every snapshot.

The guest's existing creative mode + map editor are **hard-disabled.** Pause menu still works (purely local UI). Settings, audio, key bindings — all local.

## Tick & snapshot model

- **Host tick rate:** unchanged from offline (whatever the existing game runs at — typically tied to `requestAnimationFrame`). Internally the game state is updated however it is today.
- **Snapshot rate to guests:** 20 Hz. This is the cadence the host packages and sends a delta to the relay. Independent of the local tick rate; the broadcaster samples the latest state.
- **Snapshot shape:** per-zone, sparse delta vs last-broadcast state, with one full snapshot on join/zone-change/reconnect.
- **What's in a snapshot:** every entity that changed since last broadcast (position, HP, animation step, flags), plus the per-guest `lastSeq` map for reconciliation, plus the host tick number `t` for interpolation timing.
- **Discrete events** (death, pickup, dialogue lines, cutscene start/end, toast) ride alongside as `event` frames, not in the snapshot.

20 Hz with sparse deltas should fit comfortably in 50–100 KB/s per guest at the worst case. Way under any modern connection.

## Disconnect & reconnect

- **Guest disconnects (WS close):** server marks the guest connection as ghosted with 30 s grace. Tells host via `peer.ghosted`. Host **keeps the guest's avatar in place, frozen**. Within 30 s the guest can reconnect with the same UUID and resume; after 30 s the host receives `peer.left`, despawns the avatar, frees the slot.
- **Host disconnects (WS close):** server marks the host as ghosted with 30 s grace. Guests get `host.ghosted` and a visual indicator ("Host lagging…"). Within 30 s the host can reconnect with the same UUID and resume — the world is *still in the host's browser memory*, so the resume is literally just re-establishing the WS. After 30 s, server broadcasts `session.closed` to guests, who fall back to offline. **The host loses no progress** — their save is local and uninterrupted.
- **Server restart:** every WS drops. Hosts and guests receive `4500`. Host's game continues in its browser tab (no data loss); guests fall back to offline. Both can reconnect after the server comes back: hosts re-open with a new code, guests rejoin.

There is no host migration in v0. If the host quits permanently, the session is over.

## Persistence (v0: none)

- The server holds zero persistent state.
- The host persists their save normally (localStorage, unchanged).
- Guests **do not** persist any session state. Each session join is fresh from the host's world.
- Guests *do* keep their UUID across sessions so the host can recognize them as "the same friend" — useful for future per-friend settings, irrelevant in v0.

## Cheating posture (v0)

Out of scope, but for the record:

- A guest can lie about input seq, send garbage inputs, or flood — mitigated by per-connection rate limits at the relay and intent validation on the host (an "I shoot" intent with no ammo is silently ignored, etc., same as it would be for a malformed local input).
- A guest cannot write world state — only the host's simulation can.
- A guest can predict-then-snap to mess with their own client's rendering — only hurts themselves.
- The relay sees raw JSON in plaintext over TLS — same trust model as any WS app.

If we ever want stronger guarantees against guest tampering, the host can add range/cooldown sanity checks on incoming intents. This is a small, optional hardening; it's not necessary for friend-co-op.

## Native wrapper & local split-screen

The architecture supports desktop split-screen with **zero new networking code**: a native wrapper (Electron, Tauri, ...) bundles the Node relay and the browser client into one binary.

Modes the wrapper enables:

- **Single-window offline.** Default. No relay spawned. Identical to today's offline play, identical save namespace.
- **Single-window online.** Wrapper points the window at `wss://sneakbit.curzel.it/ws`. Same flow as the browser client at <https://curzel.it/sneakbit-html>.
- **Local split-screen co-op.** Wrapper spawns the relay on `127.0.0.1` and a free port, then opens 2–4 webviews — one with `?host=1`, the rest with `?join=CODE&server=ws://127.0.0.1:<port>`. The host webview auto-`host.open`s and the others auto-`guest.join` with the generated code. RTT is sub-millisecond; everything else is identical to remote sessions.

This makes the in-browser version, the standalone desktop binary, and a future Steam release converge on the same networking surface. The relay is the only thing that needs to be portable to Node-on-desktop, and it already is (vanilla `node:http`, no native deps).

Mobile wrappers (iOS/Android) get single-window offline + single-window online. Split-screen on a phone makes no sense; on a tablet it's deferred.

# Wire protocol

## Transport

- **WebSocket.** TLS in production (`wss://sneakbit.curzel.it/ws`), plain in dev (`ws://localhost:8090/ws`).
- **Framing:** one JSON object per WebSocket frame. UTF-8.
- **Endpoint:** `/ws`. The dev server may expose it as `/`; the production nginx vhost mounts `/ws` explicitly with WebSocket upgrade headers.

### WebRTC upgrade path (shipped)

Once the WS relay works end-to-end, game traffic (snapshots, deltas,
events, inputs) moves to a WebRTC `RTCDataChannel` between host and each
guest. The relay only forwards offer/answer/ICE candidates between the
two endpoints (one `webrtc.signal` op, opaque payload). Wire shapes for
game ops are unchanged — only the transport flips.

Topology: the **guest is the initiator** for every channel. The guest
creates the data channel and ships the offer; the host listens for
offers from each peer and answers in kind. The host therefore needs no
prior knowledge of who's joining — `peer.joined` is enough to allocate a
channel slot. Once every relevant DC is `open`, the client's send-side
interceptor (`js/webrtcTransport.js`) routes `input`/`snapshot`/`delta`/
`event` over the data channels; the WS path stays warm for signaling,
lifecycle frames, and pre-DC traffic, and is the fallback if WebRTC
fails to negotiate.

NAT traversal:
- **STUN** — a list of free public servers (Google + Cloudflare) ships
  in `js/webrtcChannel.js` (`DEFAULT_STUN_SERVERS`). Enough for the
  vast majority of consumer NATs.
- **TURN** — the relay exposes `GET /turn-credentials` which returns a
  short-lived HMAC-SHA1 credential matching coturn's `use-auth-secret`
  (TURN REST API, draft-uberti-rtcweb-turn-rest-00). When the relay's
  `TURN_SECRET` + `TURN_URLS` env vars are unset, the endpoint returns
  503 and the client uses STUN only — TURN is best-effort. Operator
  runbook for self-hosting coturn on the VPS lives in `deploy.py`
  beside `TURN_ENV_FILE`.

Compression: the WS path negotiates RFC 7692 `permessage-deflate` with
`no_context_takeover` on both sides — per-message sync compression
gives ~60 % shrinkage on JSON deltas with no streaming state to
manage. Browsers always offer the extension; the relay opts in by
echoing it back during the upgrade. WebRTC's SCTP transport handles
its own framing, so no extension applies there.

## Versioning

Same shape as the other two docs. Every connection negotiates a single `protocol` integer.

- Client opens the WS and sends `hello` with the version it speaks.
- Server matches → responds with `welcome`.
- Below the server's `minProtocol` → server responds with `obsolete` and closes (code 4001). Client must reload.

## Connection lifecycle

```
1. Client opens WS
2. C → S: hello
3. S → C: welcome (or obsolete + close)
4. Role-specific handshake:
     host:  C → S: host.open
            S → C: host.opened {code}
     guest: C → S: guest.join {code}
            S → C: guest.joined {sessionId, hostName}  or  guest.joinFailed
            S → host: peer.joined {playerId, name}
5. Steady state:
     host → S → guests:  snapshot, event.*
     guest → S → host:   input
     either → S → other: ping/pong
6. Either side closes the WS
     - Server-initiated closes carry a 4xxx code
     - Guest close: host enters 30s grace then despawns the guest's avatar
     - Host close: server enters 30s grace; if host doesn't reconnect, session.closed broadcast
7. Reconnect: another WS open + hello with the same UUID, then same role handshake
     - Within 30s for the same session role: resume in place
     - After 30s: must re-open (host) or re-join (guest)
```

## Message catalogue

Every message has an `op` discriminant. `C →` means client → server, `S →` means server → client. Frames the server forwards as relay are noted explicitly.

### `hello` (C →)
```jsonc
{
  "op": "hello",
  "protocol": 1,
  "uuid": "8a1c1d2e-3b4f-4c5d-9e6f-7a8b9c0d1e2f",
  "client": "sneakbit-html"
}
```

### `welcome` (S →)
```jsonc
{
  "op": "welcome",
  "protocol": 1,
  "playerId": "p_a3f9b1",
  "name": "Player-a3f9"
}
```

### `obsolete` (S →)
```jsonc
{"op":"obsolete","minProtocol":2,"message":"please reload"}
```

### `host.open` (C →) / `host.opened` (S →)
```jsonc
{"op":"host.open"}

{"op":"host.opened","sessionId":"sess_8c3f12","code":"K7MJ2","maxGuests":3}
```

### `host.close` (C →)
```jsonc
{"op":"host.close"}
```
Server replies with `session.closed` to all guests, frees the session.

### `host.kick` (C →)
```jsonc
{"op":"host.kick","playerId":"p_b1d2e3"}
```
Sent by the host to eject a specific guest. Relay validates the sender's
role (`host`) and that `playerId` is a guest in the same session. On
success: relay closes the kicked guest's WS with code **4005** and fans
`peer.left {playerId, reason: "kicked"}` to the host and remaining
guests. The kicked guest's client does **not** auto-reconnect (4005 is
in the no-reconnect family); they fall back to offline and must
explicitly re-join. There is no host-side kick list — a kicked guest
can immediately re-attempt `guest.join` with the same code, and it's
the host's responsibility to kick again if desired.

### `guest.join` (C →) / `guest.joined` / `guest.joinFailed` (S →)
```jsonc
{"op":"guest.join","code":"K7MJ2"}

{"op":"guest.joined","sessionId":"sess_8c3f12","hostName":"Player-a3f9","peers":[{"playerId":"p_b1d2e3","name":"Player-b1d2"}]}

{"op":"guest.joinFailed","reason":"not_found"|"full"|"host_offline"}
```

### `guest.leave` (C →)
```jsonc
{"op":"guest.leave"}
```
Server tells host via `peer.left`.

### `peer.joined` (S → host)
```jsonc
{"op":"peer.joined","playerId":"p_b1d2e3","name":"Player-b1d2"}
```

### `peer.left` (S → host + remaining guests)
```jsonc
{"op":"peer.left","playerId":"p_b1d2e3","reason":"leave"|"disconnect"|"timeout"|"kicked"}
```
Fanned to the host and to every remaining guest in the session, so all
clients can despawn the departed peer's avatar in lockstep.

### `peer.ghosted` (S → host) / `host.ghosted` (S → guests)
```jsonc
{"op":"peer.ghosted","playerId":"p_b1d2e3"}     // host: guest disconnected, 30s grace
{"op":"host.ghosted"}                            // guests: host disconnected, 30s grace
```

### `session.closed` (S → guests)
```jsonc
{"op":"session.closed","reason":"host_quit"|"host_timeout"|"server_restart"}
```

### `input` (C guest → S → host)
```jsonc
{
  "op": "input",
  "seq": 12345,
  "t": 1717423424123,                  // guest wall-clock ms; host uses for telemetry only
  "intent": "moveUp"|"moveDown"|"moveLeft"|"moveRight"|"stopMove"|"interact"|"shoot"|"melee"
}
```
Server stamps `from: <guestPlayerId>` before forwarding. Rate limit 30/sec per guest; excess silently dropped.

### `snapshot` (C host → S → all guests)
Full snapshot — sent on guest join, zone change, or significant correction.
```jsonc
{
  "op": "snapshot",
  "t": 1234,
  "zoneId": 1001,
  "players": [
    {"playerId":"p_a3f9b1","x":3.0,"y":5.0,"tileX":3,"tileY":5,"direction":"down","hp":100}
  ],
  "entities": [
    {"id":4242,"species_id":1019,"frame":{"x":7,"y":2,"w":1,"h":1}}
  ],
  "lastSeq": {"p_b1d2e3": 1240, "p_c4d5e6": 887}
}
```
**Static zone data — biome tiles, construction tiles, the level's static decor — is NOT shipped over the wire.** Host and guest run the same build and load the same level JSON from disk; the guest's mirror calls its local `loadZone(zoneId)` whenever a snapshot's `zoneId` is unfamiliar. Only dynamic state (players, entities, lastSeq) crosses the wire. This keeps full snapshots in the same ~KB range as deltas and makes zone transitions cheap to broadcast.

### `delta` (C host → S → all guests)
Per-tick (20 Hz) sparse delta. Only changed fields appear.
```jsonc
{
  "op": "delta",
  "t": 1235,
  "zoneId": 1001,
  "players": [
    {"playerId":"p_b1d2e3","x":12.0,"y":7.0,"tileX":12,"tileY":7,"direction":"right","hp":95,"step":"midwalk"}
  ],
  "entities": [
    {"id":4242,"hp":12,"_open":true}
  ],
  "removed": {"entities":[9999]},
  "lastSeq": {"p_b1d2e3": 1245, "p_c4d5e6": 890}
}
```

### `event` (C host → S → guests)
Discrete one-shot events: `pickup`, `death`, `respawn`, `dialogueOpen`, `dialogueAdvance`, `dialogueClose`, `cutsceneStart`, `cutsceneEnd`, `zoneChange`, `toast`. Guest must ignore unknown kinds.
```jsonc
{"op":"event","kind":"pickup","playerId":"p_b1d2e3","speciesId":5,"amount":1}
{"op":"event","kind":"death","playerId":"p_b1d2e3"}
{"op":"event","kind":"zoneChange","zoneId":1002,"fromZoneId":1001}
```
`zoneChange` is a heads-up frame, **not** a payload carrier — the host emits it immediately *before* the new-zone `snapshot` so the guest can start fading its overlay out before the mirror swaps zones underneath it. The guest fades back in once the follow-up `snapshot` lands (or after a short timer, whichever comes first).

### `webrtc.signal` (host ↔ guest, relayed)
Opaque pass-through for WebRTC offer/answer/ICE candidates. The relay
never inspects `payload`; it routes by `to`/`from` (host fan-in for
unaddressed guest frames, host fan-out to the named guest for outbound
frames). When the resulting `RTCDataChannel` is `open`, game traffic
(snapshot/delta/event/input) moves off the WS — see
"WebRTC upgrade path" above.
```jsonc
{"op":"webrtc.signal","to":"p_b1d2e3","payload":{"kind":"offer","sdp":"v=0..."}}
{"op":"webrtc.signal","to":"p_a3f9b1","payload":{"kind":"answer","sdp":"v=0..."}}
{"op":"webrtc.signal","payload":{"kind":"ice","candidate":{"candidate":"candidate:...","sdpMid":"0","sdpMLineIndex":0}}}
```
Guest frames omit `to` — destination is always the host. The relay
stamps `from: <playerId>` on every forwarded frame. Rate limit: 10/s
per connection (the "other" bucket).

### `ping` (either →) / `pong` (S →)
Heartbeat. Server expects a `ping` at least every 30 seconds; missing pings for 60 seconds cause a close with code 4002.

## Close codes

| Code | Meaning | Client action |
|---|---|---|
| `1000` | Normal closure | Show "Disconnected" toast |
| `4001` | Obsolete protocol | Force a `location.reload()` |
| `4002` | Idle timeout (no pings) | Auto-reconnect once, then show "Disconnected" |
| `4003` | UUID conflict | Show "Already playing in another tab" |
| `4004` | Rate-limit ban | Show "Disconnected — too many messages" |
| `4005` | Kicked by host | Show "You were removed from the session"; **do not** auto-reconnect; `switchRole("offline")` |
| `4500` | Internal server error / restart | Show "Server error — reconnecting…" + auto-reconnect after 3 s |

## Rate limits

- Inputs (guest): 30/sec per connection. Excess silently dropped.
- Snapshots/deltas (host): 30/sec per connection (i.e. up to 50 % above the nominal 20 Hz to allow bursts on zone change). Excess silently dropped.
- All other ops: 10/sec per connection.
- Severe violations (1000+ msgs in 10 s) → 4004 close, same UUID can reconnect after 60 s.

## Reconnection

- On WS close, client backs off (1s, 2s, 4s, 8s, capped 30s) and re-opens.
- On reopen, sends `hello` with the same UUID.
- Within the 30 s grace, the role-specific handshake (`host.open` or `guest.join` with the same code) restores the session in place.
- After grace expiry, it's a fresh connection — guest must re-enter the code; host gets a new code.

# Implementation order

Each phase ends with a runnable, deployable state. The host is the existing client + a network add-on; the guest is the existing renderer + a network add-on.

## Phase 0 — Foundations (this doc)

- [x] This document
- [x] Audit existing local-2P-coop infra: confirmed during Phase 5/7 — input.js now

      network co-op, and inventory/equipment/playerHealth already index by player
      index. 4-player support is a thin extension once main.js gains state.players[]

## Phase 1 — Relay skeleton
Make the server speak the protocol's connection lifecycle and session ops.

1. Add WebSocket support to `server/index.js`. Hand-roll using `node:http` upgrade + RFC 6455 framing (~100 lines). No npm deps — server stays vanilla per CLAUDE.md, and that's what lets the native wrapper bundle it without a node_modules ball.
2. Implement `hello`/`welcome`, `ping`/`pong`, close codes.
3. Implement `host.open`/`host.close`/`guest.join`/`guest.leave` with in-memory state.
4. Implement fan-out: `input` from any guest goes only to the session host; `snapshot`/`delta`/`event` from host go to all guests in the session.
5. Tests: a Node-based test client opens a host session, another joins, frames forward correctly, reconnect within grace. `tests/server.session.test.js`.

## Phase 2 — Network module on the client
Both host and guest share the same `js/net.js`: WS open, reconnect with backoff, frame routing, ping/pong.

1. Implement `js/net.js` with: `connect()`, `send(frame)`, `on(op, handler)`.
2. URL gating: `?host=1` boots in host mode, `?join=CODE` in guest mode, default offline.
3. Stub host/guest mode handlers — open the WS, do the role handshake, log the connected state.
4. Manual verify: open `?host=1` in one tab and `?join=CODE` in another; both connect, server log shows the pairing.

## Phase 3 — Snapshot broadcaster on the host
The host's existing game keeps running unchanged; we add a thin **broadcaster** that samples world state and emits snapshots/deltas at 20 Hz.

1. Implement `js/snapshotBroadcaster.js`: every 50 ms, package the changed-since-last state and send `delta`. On `peer.joined`, send a full `snapshot`.
2. Serialize the world state in a shape that matches today's local state shape (so reuse loaders).
3. Manual verify: host runs the game; a guest tab logs every snapshot received but doesn't render yet.

## Phase 4 — Guest mirror rendering (no own avatar yet)
Guest builds and renders a mirror world from host snapshots. Renders host's avatar and mobs. No interactivity.

1. Implement `js/mirrorWorld.js`: ingest `snapshot` (replace) and `delta` (merge). Hold the last two frames for interpolation.
2. Reuse the existing renderer to draw the mirror world. The renderer doesn't care that the state isn't a local tick — it just draws.
3. Add interpolation: render entities at `now - 100 ms` lerped between the two latest frames.
4. Manual verify: host walks around; guest watches host's avatar move smoothly.

## Phase 5 — Guest input + host injection
Guest sends input intents; host injects them as P2/P3/P4 inputs in its existing local-coop input pipeline.

1. On guest: capture inputs that would normally drive P1, stamp with `seq`, send via `input`.
2. On host: receive `input`, look up which P-slot this guest occupies, inject into the input layer.
3. Manual verify: guest presses arrow keys; host's local game responds; host pushes the updated state via the next delta; guest sees their own avatar move (no prediction yet — laggy but correct).

## Phase 6 — Guest prediction + reconciliation
Add client-side prediction and snap-and-replay reconciliation for the guest's own avatar.

1. Track `predictedSelf` separately from the mirror world.
2. On input: apply it to `predictedSelf` immediately *and* send to host.
3. On snapshot: read `lastSeq[selfId]`, compare predicted position at that seq vs host position. If mismatch, snap and replay unacknowledged inputs.
4. Renderer: draw `predictedSelf` for the guest's own avatar; draw mirror for everyone else.
5. Manual verify: guest movement feels instant; deliberately introduce latency (Chrome DevTools throttle) and verify movement still feels instant, with occasional snap-backs on disputed steps.

## Phase 7 — Discrete events end-to-end
1. Pickup, death, respawn, dialogueOpen/Advance/Close, cutsceneStart/End, toast: host emits `event`, guest applies to local UI (HUD, inventory, dialogue modal).
2. Manual verify: host and guest play through a normal session — combat, pickups, dialogue.

## Phase 8 — Zone transitions
1. When the guest steps on a teleporter, the host's local sim handles the transition (it's just a P2/P3/P4 stepping on a teleporter in the host's world).
2. Host emits `event:zoneChange {zoneId, fromZoneId}` as a heads-up frame, then immediately follows with a fresh full `snapshot` for the new zone. The guest fades its overlay out on the event and fades back in once the snapshot is applied.
3. Guest's mirror world is reset to the new snapshot.
4. Manual verify: host and guest teleport together; both end up in the right zone with each other visible.

## Phase 9 — Polish
1. "Host lagging" UX, "Guest disconnected" toast, "Session closed" fallback to offline.
2. Lobby UX in HTML: the Party panel grows a "Open to friends" button (host mode) and a "Join with code" field (anywhere).
3. Mobile / touch quirks in online mode.
4. Audit which features should be disabled per-mode (creative + map editor for guests; possibly also for the host while a session is open).

## Phase 10+ — Beyond
- [x] WebRTC data channel for game traffic, with the relay handling only signaling. See "WebRTC upgrade path" above. (Shipped: js/webrtcChannel.js, js/webrtcTransport.js, server `webrtc.signal` op, /turn-credentials endpoint, RFC 7692 permessage-deflate on the WS path.)

# Open questions / deferred

- **PvP:** out of scope. Same as today.
- **Chat:** out of scope for v0.
- **Real accounts (email/password, OAuth):** deferred. UUID can bind later.
- **Persistent shared worlds:** explicitly *not* the model. Each session is one host's world; quitting ends it.
- **Server persistence across restarts:** out of scope; in-memory is enough.
- **Host migration:** out of scope. Host quits → session ends. Adding migration would mean serializing the host's full world state and bootstrapping a new host from it — large project, deferred.
- **Cross-mode play (host in offline-but-sharing):** out of scope; hosting requires `?host=1`.
- **Accounts / login / signup:** explicitly out of scope. Random per-browser UUID is identity. If we ever want persistent friend lists, achievements, or cross-device save, an account layer can bind to an existing UUID retroactively.
- **Compression / binary frames:** maybe per-message deflate later; not needed at 20 Hz with sparse deltas.
- **Determinism / lockstep:** not used. The host runs the only sim; nothing else needs to be deterministic.
- **WebRTC vs WS for game traffic:** WS for v0 (simpler), WebRTC as a Phase 10 optimization. Either way the wire shapes stay the same.
- **Slot reassignment on guest reconnect:** if guest A drops, guest B joins (taking slot P2), then A reconnects within grace — what slot does A get? Default: A gets P2 back, B is bumped to P3 if free, else rejected. To revisit if it's actually a problem in practice.
